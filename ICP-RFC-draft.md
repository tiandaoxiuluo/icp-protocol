# ICP-0.1: Information Credibility Protocol

**A Minimal Open Standard for Verifying Human-Created Content**

## Abstract

The proliferation of large language models has made it increasingly difficult to distinguish AI-generated content from human-authored content. Existing approaches—platform watermarks, copyright notices, and steganographic techniques—either depend on centralized authorities or are easily circumvented.

This document specifies the Information Credibility Protocol (ICP), a minimal open standard that uses asymmetric cryptography to produce a verifiable, tamper-evident proof of authorship for any text content. The protocol requires no trusted third party. Verification relies solely on standard cryptographic primitives available in any modern programming environment.

## 1. Problem Statement

### 1.1 The Indistinguishability Problem

Current language models can produce text that is stylistically, grammatically, and semantically indistinguishable from human writing. At scale, this creates a verification gap: there is no reliable, lightweight mechanism for a reader to determine whether content was authored by the claimed party or generated by a machine.

### 1.2 Limitations of Existing Approaches

| Approach | Weakness |
|---|---|
| Platform-level watermarks | Depends on platform compliance; removed if content is copied |
| Copyright notices | No cryptographic binding; trivially forged |
| AI detection classifiers | High false-positive rate; adversarially bypassable |
| Steganographic watermarks | Destroyed by format conversion, compression, or transcription |
| Blockchain timestamping | High complexity; requires on-chain transaction infrastructure |

### 1.3 Core Requirement

What is needed is a scheme with the following properties:

- **Authorship binding** — content is cryptographically linked to a specific key pair
- **Tamper evidence** — any modification to the content invalidates the proof
- **Decentralized verification** — verification does not require querying any server
- **Minimal footprint** — implementable in under 100 lines of code in any language

---

## 2. Design Goals

**G1 — Simplicity.** The protocol must be implementable from scratch by a competent developer in under two hours.

**G2 — No trusted third party.** Verification must be possible using only the signed statement itself and standard cryptographic libraries.

**G3 — Human-readable format.** Signed statements are JSON and can be inspected without special tooling.

**G4 — Forward compatibility.** The protocol version field allows future algorithm upgrades without breaking existing implementations.

**G5 — Privacy-neutral.** The protocol makes no assumptions about identity. A public key is the only identifier required.

## 3. Protocol Overview

### 3.1 Roles

```
Author
  └─ holds a private key
  └─ produces a Signed Statement

Verifier
  └─ holds the Signed Statement
  └─ validates signature using the embedded public key
```

### 3.2 Signing Flow

```
Input text
    │
    ├─ encode as UTF-8 bytes
    │
    ▼
SHA-256 digest
    │
    ▼
RSA-PKCS1v15 sign with private key
    │
    ▼
Base64-encode signature
    │
    ▼
Assemble Signed Statement (JSON)
    {
      protocol, content, timestamp,
      signature, public_key
    }
```

### 3.3 Verification Flow

```
Signed Statement (JSON)
    │
    ├─ extract: content, signature, public_key, timestamp
    │
    ▼
Decode Base64 signature
    │
    ▼
RSA-PKCS1v15 verify:
    public_key + SHA-256(content.encode("utf-8")) + signature
    │
    ├─ PASS → statement is authentic and unmodified
    └─ FAIL → statement has been tampered with or key is wrong
```

## 4. Specification

### 4.1 Signed Statement Format

A Signed Statement is a UTF-8 encoded JSON object with the following top-level fields:

```json
{
  "protocol":   "ICP-0.1",
  "content":    "<original text, UTF-8>",
  "timestamp":  "<ISO 8601 UTC datetime>",
  "signature":  "<Base64-encoded RSA-SHA256 signature>",
  "public_key": "<PEM-encoded RSA public key>"
}
```

All fields are **required**. Additional fields MAY be included by implementations but MUST be ignored by verifiers operating under this version.

### 4.2 Field Definitions

| Field | Type | Description |
|---|---|---|
| `protocol` | string | Protocol identifier. MUST be `"ICP-0.1"` for this version. |
| `content` | string | The original text content, preserved exactly as signed. |
| `timestamp` | string | UTC time of signing in format `YYYY-MM-DDTHH:mm:ss.SSSSSSZ` |
| `signature` | string | Base64 (standard, padded) encoding of the RSA-PKCS1v15 signature over `SHA-256(content.encode("utf-8"))` |
| `public_key` | string | PEM-encoded SubjectPublicKeyInfo (begins with `-----BEGIN PUBLIC KEY-----`) |

### 4.3 Cryptographic Primitives

- **Hash function:** SHA-256
- **Signature scheme:** RSA with PKCS#1 v1.5 padding (RSASSA-PKCS1-v1_5)
- **Minimum key size:** 2048 bits
- **Recommended key size:** 2048 bits (standard) or 4096 bits (long-term use)
- **Public key encoding:** SubjectPublicKeyInfo, PEM format

### 4.4 Timestamp

- MUST use Coordinated Universal Time (UTC)
- MUST conform to ISO 8601: `YYYY-MM-DDTHH:mm:ss.SSSSSSZ`
- Verifiers SHOULD check that the timestamp is not unreasonably far in the future

### 4.5 Content Normalization

The `content` field MUST be signed as-is, without normalization (no trimming, no line-ending conversion). Implementations that alter whitespace before signing will produce statements that cannot be verified by compliant implementations that display the content differently.

## 5. Security Considerations

### 5.1 Private Key Security

The security of the scheme depends entirely on the secrecy of the private key.

- Private keys MUST NOT be stored in plaintext in publicly accessible locations.
- Hardware security modules (HSMs) or hardware tokens (e.g., FIDO2 keys) are recommended for keys used in high-stakes contexts.
- Key compromise must be communicated out-of-band. The protocol does not include a revocation mechanism at this version.

### 5.2 Public Key Authenticity

The public key embedded in a Signed Statement cannot be independently verified as belonging to a specific real-world identity. This is a deliberate design choice to avoid a trusted third party dependency.

Implementations that require identity binding (e.g., linking a public key to an email address or domain) should consider integration with:

- Domain-based key publication (DNS TXT records)
- Decentralized Identifiers (DIDs) per W3C specification

### 5.3 Timestamp Trust

Timestamps are self-reported and not independently verified. An author could sign content with an incorrect timestamp.

Mitigations:
- RFC 3161 Trusted Timestamp Authority (TSA) integration (see Section 7)
- Witness-based timestamping via append-only public logs

### 5.4 Repudiation

Because private key possession is the sole proof of authorship, a signer can claim the key was compromised after signing. This protocol does not solve the general non-repudiation problem. For use cases requiring legal non-repudiation, additional mechanisms (e.g., TSA countersignatures, witness logs) are necessary.

### 5.5 Algorithm Longevity

RSA-2048 is currently considered secure against classical computing adversaries. The `protocol` version field is provided specifically to allow a migration path when stronger primitives are required.

Planned migration: ICP-0.2 will evaluate CRYSTALS-Dilithium or FALCON as post-quantum alternatives.

## 6. Use Cases

### 6.1 Editorial Authenticity

A journalist or blogger signs their article before publication. Readers can verify the byline is not forged and the text has not been altered since signing.

### 6.2 Academic Priority

A researcher signs a draft before submission to establish a timestamped record of priority. This supplements (but does not replace) formal peer review processes.

### 6.3 Contractual Records

Parties to an informal agreement sign the agreed terms. The resulting statements serve as tamper-evident records without requiring a notary.

### 6.4 Source Material Provenance

A news aggregator or fact-checking service can store signed statements alongside republished content to retain an auditable link to the original source.

---

## 7. Future Work

### 7.1 Trusted Timestamp Integration

Integration with RFC 3161 Time-Stamp Protocol (TSP) to replace self-reported timestamps with countersignatures from an independent Timestamp Authority.

### 7.2 Decentralized Identity (DID) Binding

Association of ICP public keys with W3C DID documents, enabling verifiers to resolve a human-readable or institutional identity without a centralized registry.

### 7.3 Selective Disclosure

A mechanism to prove properties of signed content (e.g., "this document was authored before date X") without revealing the full content, using zero-knowledge techniques.

### 7.4 Post-Quantum Cryptography Migration

A planned ICP-0.2 version will evaluate replacement of RSA with a NIST-standardized post-quantum signature scheme (e.g., ML-DSA / CRYSTALS-Dilithium).

### 7.5 Statement Registry

An optional, open, append-only log where signed statements can be published for public indexing—analogous to Certificate Transparency logs for TLS certificates.

## 8. Reference Implementation

A reference implementation in Python is available at:

> [GitHub repository URL — to be published]

**Dependencies:** `pycryptodome >= 3.0`

**Minimal signing example:**

```python
from Crypto.PublicKey import RSA
from Crypto.Signature import pkcs1_15
from Crypto.Hash import SHA256
import base64, json
from datetime import datetime, timezone

def sign(content: str, private_key_pem: bytes) -> dict:
    key = RSA.import_key(private_key_pem)
    h = SHA256.new(content.encode("utf-8"))
    sig = pkcs1_15.new(key).sign(h)
    return {
        "protocol":   "ICP-0.1",
        "content":    content,
        "timestamp":  datetime.now(timezone.utc).isoformat(),
        "signature":  base64.b64encode(sig).decode(),
        "public_key": key.publickey().export_key().decode()
    }

def verify(statement: dict) -> bool:
    pub = RSA.import_key(statement["public_key"])
    sig = base64.b64decode(statement["signature"])
    h   = SHA256.new(statement["content"].encode("utf-8"))
    try:
        pkcs1_15.new(pub).verify(h, sig)
        return True
    except (ValueError, TypeError):
        return False
```
