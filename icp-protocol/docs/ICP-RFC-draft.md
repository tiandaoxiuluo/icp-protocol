# ICP-0.1: Information Credibility Protocol
## A Minimal Standard for Human-Created Content Verification in the Age of AI

**状态：** 草案 v0.1  
**日期：** 2026-04-23  
**作者：** 刘家麒  
**协议版本：** ICP-0.1  

---

## 摘要（Abstract）

大型语言模型（LLM）的爆发式普及，使得AI生成内容与人类原创内容之间的界限日益模糊。在这一背景下，信息的真实性和来源可信度面临前所未有的挑战。

ICP（Information Credibility Protocol，信息可信协议）是一个极简的开放标准，旨在为任何文本内容提供密码学级别的"人类创作证明"。通过RSA非对称加密和SHA-256哈希算法，ICP为每段文字生成唯一的数字签名，任何人都可以验证该内容是否被篡改，以及其发布者的身份。

本协议不依赖任何中心化机构，任何人均可自由使用和实现。

---

## 问题陈述（Problem Statement）

### 1.1 AI生成内容的泛滥

当前，LLM可以在数秒内生成看似合理的新闻文章、学术摘要、个人陈述甚至法律文件。这些内容与真人撰写的内容在形式上几乎无法区分。

### 1.2 虚假信息的危害

- **政治层面**：选举期间大规模传播AI生成的假新闻
- **社会层面**：虚假健康信息、诈骗内容的快速扩散
- **个人层面**：AI生成内容被冒充为个人作品（学术、创意作品）

### 1.3 现有方案的局限

- 平台水印：依赖平台方，可被轻易去除
- 版权声明：无加密验证，无法防止篡改
- 数字水印（隐写）：容易被压缩、截图破坏

## 协议概述（Protocol Overview）

### 3.1 核心角色

```
发布者（Publisher）
    │
    │ 1. 撰写内容
    ▼
[签名引擎]
    │ 2. 用私钥对内容进行RSA-SHA256签名
    ▼
[可信声明（Signed Statement）]
    │ 3. 附带时间戳、公钥、签名
    ▼
读者/验证者（Reader/Verifier）
    │ 4. 用公钥验证签名完整性
    ▼
[验证结果：有效 / 被篡改 / 无效]
```

### 3.2 签名流程

```
原始内容 ──→ SHA-256哈希 ──→ RSA-2048加密 ──→ 签名（Base64）
         └──────────────────────────────┘
              与内容+时间戳+公钥组合
              生成可信声明（Statement）
```

### 3.3 验证流程

```
接收声明 ──→ 提取公钥 ──→ 用公钥解密签名
    │
    │ 比对
    ▼
计算内容SHA-256 ──→ 比对哈希值
    │
    ▼
签名匹配？──否──→ 验证失败（内容被篡改）
    │
   是
    ▼
签名匹配且哈希一致 → 验证通过
```

## 技术规范（Specification）

### 4.1 可信声明格式（JSON Schema）

```json
{
  "protocol": "ICP-0.1",
  "content": "<原始文本内容>",
  "timestamp": "<ISO 8601格式UTC时间>",
  "signature": "<Base64编码的RSA-SHA256签名>",
  "public_key": "<PEM格式的RSA公钥>"
}
```

### 4.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| protocol | string | 是 | 协议标识符，当前为 "ICP-0.1" |
| content | string | 是 | 原始文本内容，UTF-8编码 |
| timestamp | string | 是 | ISO 8601格式：`YYYY-MM-DDTHH:mm:ss.ffffffZ` |
| signature | string | 是 | Base64编码的PKCS#1 v1.5签名 |
| public_key | string | 是 | PEM格式的RSA公钥（-----BEGIN PUBLIC KEY-----） |

### 4.3 签名算法

- **哈希算法**：SHA-256
- **签名算法**：RSA-PKCS#1 v1.5
- **密钥长度**：2048位（推荐）/ 4096位（高安全场景）

### 4.4 时间戳

- 必须使用UTC时间
- 格式：`YYYY-MM-DDTHH:mm:ss.SSSSSSZ`
- 验证者应检查时间戳合理性（防止重放攻击）

### 4.5 私钥格式

```
-----BEGIN RSA PRIVATE KEY-----
<Base64编码的DER格式私钥>
-----END RSA PRIVATE KEY-----
```

## 安全考量（Security Considerations）

### 5.1 私钥保管

- 私钥必须妥善保管，建议使用硬件密钥（如YubiKey）
- 不建议在云端存储私钥，或至少启用多因素认证
- 私钥泄露后，应立即生成新密钥对并公告旧公钥作废

### 5.2 公钥可信性

- 公钥嵌入声明本身，验证者可自行验证签名
- 但公钥本身可能被替换，解决方案见"未来方向"
- 推荐配合域名/邮箱关联公钥，建立身份层

### 5.3 算法演进

- RSA-2048在当前计算能力下安全，但量子计算可能构成威胁
- 后量子算法（如CRYSTALS-Dilithium）将成为未来升级方向
- 协议预留 `protocol` 版本字段，支持平滑升级

### 5.4 否认风险

- 私钥持有者可签署任何内容，之后声称密钥被盗
- 解决方案：结合时间戳服务（TSA）证明签署时间

### 5.5 隐私

- 声明内容以明文存储，任何人都可读取
- 如需隐私保护，应在签名前对内容加密

## 应用场景（Use Cases）

### 6.1 内容原创证明

博主、记者在发布文章时附上ICP声明，证明内容由本人创作。

### 6.2 合同/协议签署

在电子合同中嵌入ICP声明，防止事后篡改。

### 6.3 学术诚信

论文预印本附ICP声明，证明特定版本的内容由特定作者在特定时间发布。

### 6.4 供应链溯源

企业发布声明证明文件的真实性和发布时间。

## 未来方向（Future Work）

### 7.1 去中心化身份（DID）集成

将ICP声明与去中心化身份系统（如DID/VC）结合，实现自主主权身份。

### 7.2 时间戳权威（TSA）集成

引入可信时间戳服务，防止签名者事后否认。

### 7.3 后量子签名

在协议升级时（ICP-0.2+）引入基于格的后量子签名算法。

### 7.4 零知识证明（ZKP）扩展

允许声明者在不暴露内容的情况下证明"我知道这段内容"。

### 7.5 公钥基础设施（PKI）整合

探索与现有PKI体系（如Let's Encrypt）的整合可能。

### 7.6 协议注册表

建立ICP协议声明的公开注册表，允许第三方索引和搜索可信声明。
